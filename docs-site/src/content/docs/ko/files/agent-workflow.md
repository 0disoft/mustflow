`.mustflow/docs/agent-workflow.md`는 에이전트가 저장소에서 어떤 순서로 읽고, 수정하고, 검증해야 하는지 정리하는 문서입니다.

## 어디에 쓰이나

`AGENTS.md`가 짧은 첫 진입 규칙이라면, 이 파일은 그 규칙을 실제 작업 흐름으로 풀어주는 공통 정책 문서입니다.

에이전트는 작업을 시작할 때 `AGENTS.md` 다음에 이 파일을 읽고, 명령 실행, 입력 안정성, 맥락 압축, 수정 범위, 실패 대응의 기본 기준을 확인합니다.

## 구성요소

- `문서의 역할`: 이 파일이 담당하는 정책 범위를 설명합니다.
- `기준 문서 역할`: 어떤 mustflow 파일이 규칙, 문맥, 절차, 선호값, 생성 탐색 결과를 담당하는지 설명합니다.
- `기준 문서와 읽기 흐름`: 에이전트가 어떤 파일을 어떤 순서로 읽는지 정합니다.
- `프로젝트 맥락`: `.mustflow/context/INDEX.md`와 작업별 맥락 파일을 언제 읽을지 설명합니다.
- `스킬 활성화`: 에이전트가 작업별 스킬 절차를 언제 고르고 읽어야 하는지 정합니다.
- `작업 시작 전 확인`: 변경 파일, 보호 경로, 명령 의도, 필요한 스킬을 확인하게 합니다.
- `입력 안정성 정책`: 자주 바뀌는 정보를 필수 읽기 문서 앞부분에서 분리합니다.
- `프롬프트 캐시와 호스트 맥락 구성`: 안정, 작업, 변동 맥락을 나누되 캐시 상태를 권위 출처로 보지 않게 합니다.
- `지침 새로고침 정책`: 긴 세션에서 mustflow 지침을 언제 다시 읽을지 정합니다.
- `맥락 압축 정책`: 최신 파생 맥락, 중간 요약, 장기 요약의 경계와 권위 순서를 설명합니다.
- `하네스와 런타임 표면`: 저장소 작업 흐름 파일이 선택형 런타임, 조정, 작업 항목, 어댑터, 하네스 표면으로 확장되는 방식을 설명합니다.
- `장기 작업 단계`: 계획, 작업, 검증, 판정, 인계를 분리합니다.
- `검증 약화 방지`: 완료처럼 보이게 만들기 위해 검증을 약화하지 못하게 합니다.
- `테스트 관련성 정책`: 테스트를 현재 동작 계약에 맞게 추가, 수정, 삭제, 보고하게 합니다.
- `선호값 해석 정책`: `preferences.toml`의 언어, 서식, 커밋, 로그 기본값을 어떻게 적용할지 설명합니다.
- `Git 행동 정책`: 자동 스테이징, 자동 커밋, 자동 푸시를 금지하고 커밋 메시지 추천을 보고 규칙으로 분리합니다.
- `버전 영향 정책`: 버전 판올림이 필요한지 확인하고 제안하되, 자동 버전 수정은 기본적으로 막습니다.
- `명령 실행 정책`: `commands.toml`에 선언된 단발성 명령 의도만 실행하게 합니다.
- `수정 정책`: 요청과 직접 관련된 파일만 수정하게 합니다.
- `검증 정책`: 변경 뒤 어떤 검증 의도를 확인해야 하는지 설명합니다.
- `실패 대응 정책`: 실패한 명령 의도, 작업 디렉터리, 종료 코드, 핵심 오류를 남기게 합니다.
- `보안 및 비밀정보 정책`: 토큰, 개인 키, 실제 환경 파일 값을 노출하지 않게 합니다.
- `문서 흐름 유지`: 규칙, 명령, 스킬, 보호 경로가 바뀔 때 갱신할 파일을 안내합니다.

## 기준 문서 역할

설치된 작업 흐름은 권한을 좁게 나눕니다.

- `AGENTS.md`: 첫 진입점이며 반드시 지켜야 하는 짧은 저장소 규칙입니다.
- `.mustflow/docs/agent-workflow.md`: 공통 작업 정책입니다. `AGENTS.md`를 풀어 설명하지만 실행 명령은 정의하지 않습니다.
- `.mustflow/config/mustflow.toml`: 기계가 읽는 작업 흐름 설정입니다.
- `.mustflow/config/commands.toml`: 설정된 의도를 통해 프로젝트 명령 실행을 허용하는 유일한 기준입니다.
- `.mustflow/config/preferences.toml`: 낮은 권위의 저장소 기본값이며 권한이 아닙니다.
- `.mustflow/context/INDEX.md`: 선택형 작업 문맥을 고르는 라우터입니다.
- `.mustflow/context/PROJECT.md`: 코드, 테스트, 명령 계약, 설정된 정책보다 낮은 프로젝트 사실과 모르는 점입니다.
- `.mustflow/skills/router.toml`: 첫 절차 선택을 위한 안정적인 압축 라우트 범주와 fallback 규칙입니다.
- `.mustflow/skills/routes.toml`: 압축 router만으로 부족할 때 사용하는 전체 라우트 메타데이터입니다.
- `.mustflow/skills/INDEX.md`: 상세 선택과 라우트 유지보수를 위한 확장 라우트 표입니다.
- `.mustflow/skills/<name>/SKILL.md`: 경계가 정해진 반복 절차이며 명령 실행 권한을 줄 수 없습니다.
- `REPO_MAP.md`: 설정된 `repo_map` 의도나 `mf map`으로 갱신하는 생성 앵커 지도입니다.

## 명령 실행 정책

실행 가능한 명령 의도의 기준 자료는 `.mustflow/config/commands.toml`입니다.

에이전트는 모든 실행 관문을 만족하는 명령 의도만 실행합니다. 조건은 `status = "configured"`, `lifecycle = "oneshot"`, `run_policy = "agent_allowed"`, `stdin = "closed"`, 양의 정수 `timeout_seconds`, `argv` 또는 `mode = "shell"`과 `cmd`로 선언된 명령, 현재 mustflow 루트 안의 `cwd`입니다. 의도가 없거나 `unknown`, `not_applicable`, `manual_only`, `disabled`이면 대체 명령을 추측하지 않고 건너뛴 이유를 보고합니다.

새 설정에서는 `manual_only`를 명령 의도의 `status` 값으로 사용합니다. `run_policy = "manual_only"`는 기존 설정 호환을 위해 허용될 수 있지만, 새 템플릿에서는 생성하지 않습니다.

`configured` 의도는 가능하면 `argv` 배열을 사용합니다. 셸 문법이 꼭 필요할 때만 `mode = "shell"`과 `cmd`를 사용합니다.

`server`, `watch`, `interactive`, `browser`, `background` 생명주기는 직접 실행하지 않습니다. 개발 서버, 감시 모드, 브라우저 UI, 백그라운드 프로세스는 단발성 검증 명령으로 취급하지 않습니다.

`mf run <intent>`를 사용할 수 있으면 단발성 명령은 그 실행기를 우선 사용합니다.

검증 선택은 위험 기반입니다. 바뀐 표면을 덮는 관련 테스트, 빠른 검사, 빌드, 문서 전용
검증 의도가 설정되어 있으면 전체 테스트보다 우선하고, 더 좁은 의도가 없으면 느린 전체
테스트를 조용히 기본값으로 삼지 말고 그 공백을 보고합니다.

`mf run`은 마지막 실행 결과를 `.mustflow/state/runs/latest.json` 실행 기록으로 남깁니다.
자동화나 최종 보고에서 실행 근거가 필요하면 `mf run <intent> --json`을 사용합니다.
이 실행 기록은 특정 실행의 증거이며, 명령 정의의 기준 자료는 계속 `commands.toml`입니다.

호스트 셸은 명령을 실행할 수 있지만, 프로젝트 명령을 직접 실행한 결과가 곧 mustflow 검증 근거가 되지는 않습니다.
`mf run`을 우회한 명령 출력은 사용자가 명시적으로 수동 우회를 승인했고 최종 보고에 mustflow 실행 기록이 없음을 밝힌 경우가 아니라면 낮은 신뢰도의 참고 정보로만 봅니다.

`mf context --json`은 현재 루트의 읽기 순서, 명령 의도, 기능 표면, 마지막 실행 요약을
빠르게 보여주는 읽기 전용 색인입니다. 이 출력은 실제 문서와 설정 파일 읽기를 대신하지
않으며, 프로젝트 테스트나 빌드 명령은 계속 `commands.toml`의 의도 계약을 따릅니다.

`mf doctor` 또는 `mf doctor --json`은 수정 전 현재 mustflow 루트의 설치 상태, 검사 결과,
실행 가능한 명령 의도, 다음 단계를 함께 보는 읽기 전용 진단 명령입니다. 파일을 쓰지
않으므로 에이전트가 첫 방향을 잡을 때 사용할 수 있습니다.

mustflow 문서 흐름, 스킬, 명령 계약, 저장소 지도 생성 규칙을 바꾼 뒤에는 가능하면
`mf check --strict`를 실행합니다. 이 검사는 스킬 색인과 스킬 본문의 일치 여부, 스킬 문서의 원시 셸 명령 블록,
`REPO_MAP.md`의 안정적인 생성 메타데이터와 변동성 높은 값, 명령 출력 제한, 보존 정책, 생성 파일 크기,
원시 JSONL 로그 흔적, 최신 실행 기록 형식을 추가로 확인합니다.

## 스킬 활성화

스킬은 자율 실행 도구가 아니라 작업 절차 문서입니다. 스킬을 활성화한다는 말은
현재 명령 계약 안에서 맞는 `.mustflow/skills/<name>/SKILL.md`를 읽고 그 절차를
따른다는 뜻입니다.

작업 시작 시점과 첫 수정 전, 에이전트는 사용자 요청과 예상 변경 파일을
`.mustflow/skills/router.toml`의 범주 신호와 비교합니다. 압축 router만으로 부족하거나
스킬 라우팅을 수정하거나 상세 라우트 메타데이터가 필요하거나 라우트 확신도가 애매하면
`.mustflow/skills/routes.toml`을 읽습니다. 전체 라우트 메타데이터만으로도 부족하거나 확장
라우트 표를 수정하거나 사람이 읽을 트리거 근거가 필요하면 `.mustflow/skills/INDEX.md`를 읽습니다.
맞는 시나리오가 하나 이상이면 해당 범위를 수정하기 전에 각 `SKILL.md`를 읽습니다.
명령 실패, 테스트 계약 변경, 문서 변경처럼 작업 중 새 근거가 생기면 잠시 멈추고 새로 맞는 스킬을 읽은 뒤 계속합니다.

스킬은 원시 셸 명령, 장기 실행 프로세스, 작업 범위 밖 쓰기를 허용하지 않습니다.
또한 사용자, 호스트, 저장소, 안전 규칙을 덮어쓰지 않습니다.

## 입력 안정성

공통 규칙은 이 문서에 두고, 실제 실행 계약은 `commands.toml`에 둡니다.

반복 작업 절차는 `.mustflow/skills/`의 스킬 문서로 분리하고, 스킬 문서에는 공통 정책 전문을 복사하지 않습니다.

프로젝트 방향과 도메인 약속은 `.mustflow/context/`에 둡니다. 에이전트는 작업별 맥락이
필요할 때만 `.mustflow/context/INDEX.md`를 읽고, 색인이 고른 맥락 파일만 읽습니다.

맥락 파일은 사용자 직접 지시, 현재 코드, 테스트, 명령 계약, 설정된 정책보다 낮은
권위의 참고 자료입니다. `DESIGN.md`가 있으면 UI 작업의 선택 외부 시각 디자인 앵커로
사용하고, 디자인 토큰을 `.mustflow/context/`에 중복 기록하지 않습니다.

저장소 탐색 지도는 직접 긴 문서로 유지하지 않고 생성 가능한 `REPO_MAP.md`로 분리합니다. `REPO_MAP.md`는 전체 파일 목록이 아니라 주요 앵커 파일을 모은 지도이며, 필수 읽기 순서가 아니라 넓은 탐색이 필요할 때만 읽습니다.

파일 상단에는 안정적인 생성 메타데이터만 둡니다. 생성 시각, 브랜치명, 원격 저장소 주소, 최근 변경 요약, 긴 로그처럼 자주 바뀌는 값은 두지 않습니다.

전체 대화 원문, 전체 터미널 출력, 원시 JSONL 이벤트 로그는 `.mustflow/` 안에 계속
덧붙이지 않습니다. 실행 결과는 작은 실행 기록으로 남기고, 지식 문서는 원문 로그가
아니라 요약과 출처만 남깁니다.

## 프롬프트 캐시와 호스트 맥락 구성

프롬프트 캐시는 성능 최적화용 힌트일 뿐입니다. 캐시된 지침, 요약, 검색 결과, 생성 지도는
사용자 직접 지시, 현재 파일, 현재 명령 계약, 호스트 안전 규칙, 가장 가까운 `AGENTS.md`
보다 낮은 권위를 가집니다.

호스트가 모델 입력을 조립할 때는 안정적인 저장소 지침과 압축 route kernel을 앞에 두고,
선택된 작업별 맥락을 그다음에 두며, 자주 바뀌는 상태는 마지막에 둡니다. 안정 접두부에는
`mf context --json --cache-profile stable` 결과를 사용하고, 작업 맥락에는 선택된 파일과
`cache_layer`가 `task`인 `mf search --json` 결과를 사용합니다. 현재 사용자 요청, 변경 파일,
명령 출력 끝부분, 시각 정보, 최신 실행 기록 메타데이터, `volatile` 값이 `true`인 검색 결과는
접미부에만 둡니다.

안정 접두부를 재사용하기 전에는 콘텐츠 해시를 현재 파일과 비교합니다. 해시가 바뀌었으면
파일을 다시 읽습니다. 캐시 계층 필드는 배치 힌트일 뿐이며, 명령 실행 권한을 주거나 검색
결과가 작업 규칙을 덮어쓰게 만들지 않습니다.

## 유효 규칙 구분

모든 지침을 하나의 우선순위 목록으로 합치지 않습니다. 규칙 종류별로 충돌을 판단합니다.

- 사용자 목표: 현재 사용자 직접 지시는 안전하지 않은 경우가 아니라면 작업 목표를 정합니다.
- 호스트 안전: 호스트의 승인, 샌드박스, 체크포인트, 셸 실행 제한은 계속 적용됩니다.
- 저장소 작업 규칙: 가장 가까운 `AGENTS.md`와 `.mustflow/config/*.toml`이 저장소 계약을 정합니다.
- 명령 실행: `.mustflow/config/commands.toml`이 프로젝트 명령 계약입니다.
- 검증 근거: `mf run` 실행 기록과 현재 파일은 호스트 셸 직접 실행 출력보다 높은 신뢰를 가집니다.
- 문맥과 선호값: `.mustflow/context/*`, `preferences.toml`, 생성 지도는 낮은 권위의 기본값입니다.
- 세션과 캐시 상태: 호스트 요약, `.mustflow/cache/**`, `.mustflow/state/**`는 현재 파일이나 현재 사용자 지시보다 우선하지 않습니다.

허용 목록은 더 좁은 교집합으로 줄이고, 금지 행동, 승인 요구, 개인정보 규칙, 파괴적 명령 규칙은 누적합니다. 유효 규칙이 불명확하면 추측하지 말고 멈춘 뒤 충돌을 보고합니다.

## 지침 새로고침

긴 세션에서는 작업 시작 때 읽은 지침이 뒤쪽 대화와 도구 출력에 밀릴 수 있습니다.
`agent-workflow.md`는 이를 프로젝트 파일에 대화 횟수를 쓰는 문제가 아니라, 지침을
다시 확인해야 하는 지점 문제로 다룹니다.

에이전트는 첫 수정 전, 현재 명령 의도에 대한 신선한 명령 수준 새로고침이 없을 때의 명령 실행 전,
맥락 압축 뒤, `AGENTS.md` 또는 `.mustflow/**` 수정 뒤, 루트 변경 뒤, 최종 보고 전에 mustflow 지침을 다시 확인해야 합니다.

다시 읽을 파일 묶음은 `.mustflow/config/mustflow.toml`의 `[refresh.levels]`를 따릅니다.

## 맥락 압축 정책

긴 세션에서 만들어지는 압축 요약은 작업 기억을 돕는 파생 자료입니다. `agent-workflow.md`는
압축 요약이 현재 사용자 지시, 현재 코드와 설정, 명령 계약, 실행 기록보다 낮은 권위를
가져야 한다고 정합니다.

프로젝트 안에는 숨겨진 내부 추론, 비밀정보, 끝없이 커지는 전체 대화 원문을 저장하지
않습니다. 공유할 가치가 있는 내용은 출처를 가진 결정, 조사, 인계 요약으로만 남깁니다.

## 하네스와 런타임 표면

mustflow는 저장소 안의 작업 흐름 파일과 명령 실행 경계에서 시작합니다. 선택형 런타임,
조정, 작업 항목, 어댑터, 하네스 표면은 명시적이고, 경계가 있고, 검토 가능할 때 확장할 수 있습니다.

- 사고 표면: `AGENTS.md`, `agent-workflow.md`, 스킬 문서입니다.
- 실행 표면: `commands.toml`, `mf run`, 단발성 명령 생명주기입니다.
- 세션 표면: 제한된 실행 기록, 출처를 가진 압축 요약, 작은 인계 기록, 향후 작업 항목 기록입니다.
- 판정 표면: 원래 목표, 완료 기준, 변경 파일, 명령 계약, 실행 기록입니다.
- 런타임 표면: 작업자, 페르소나, 다중 실행 조정, 서비스 프로세스, 자율 반복 실행기는 생명주기, 보존, 격리, 승인, 검증 규칙이 필요합니다.

## 장기 작업 단계

장기 작업은 계획, 작업, 검증, 판정, 인계를 분리해야 합니다. 판정 단계는 작업자가
“완료”라고 말한 사실만으로 충분하다고 보지 않고, 원래 기준과 변경 파일, 실행 기록을
확인합니다.

## 검증 약화 방지

이 작업 흐름은 작업을 완료한 것처럼 보이게 만들기 위해 검증을 약화하는 일을 금지합니다.
에이전트는 실패하는 테스트 삭제, 이유 없는 단언 완화, 관련 명령 의도 건너뛰기, 실패를
피하기 위한 명령 의도 상태 변경, 구현 뒤 완료 기준 변경을 하지 않습니다.

의도한 동작이 바뀌었거나 기존 테스트가 잘못된 경우에는 테스트를 바꿀 수 있지만, 최종
보고에서 이유를 설명해야 합니다.

## 테스트 관련성 정책

테스트는 현재 동작 계약을 검증하는 장치입니다. 오래된 테스트가 기대한다는 이유만으로
삭제된 기능을 다시 구현하거나, 의도적으로 제거된 기능의 테스트를 계속 보존하지 않습니다.

테스트를 삭제하거나 단언을 약하게 만들 때는 검증을 피하려는 변경인지, 현재 계약에 맞춘
정리인지 구분해야 합니다. 확신이 없으면 삭제하지 않고 낡은 테스트 후보로 보고합니다.

## 선호값 해석 정책

`.mustflow/config/preferences.toml`은 사용자 직접 지시나 기존 파일 스타일보다 낮은 저장소별 기본값입니다.

에이전트는 이 파일을 이유로 전체 파일을 재포맷하거나, 요청과 무관한 파일을 바꾸거나, 기존 로그 문자열을 임의로 번역하지 않아야 합니다.

`preserve_existing`은 기존 관습이 보일 때 그 관습을 따른다는 뜻입니다. 기존 관습이 보이지 않는 신규 저장소에서는 각 항목의 `fallback` 값을 사용합니다.

사용자의 대화 언어를 근거로 코드 주석, 로그, 오류 메시지, 커밋 메시지 언어를 자동 결정하면 안 됩니다.

## Git 행동 정책

`git.auto_stage`, `git.auto_commit`, `git.auto_push`는 기본적으로 모두 `false`입니다.

이 값들은 저장소 선호값이지 권한이 아닙니다. 사용자 직접 지시, `.mustflow/config/commands.toml`, `.mustflow/config/mustflow.toml`의 승인 정책보다 우선하지 않습니다. 특히 `git.auto_commit = true`는 푸시 권한을 뜻하지 않으며, `git.auto_push = true`는 `mf init`으로 설정할 수 없습니다.

사용자 요청 없이 스테이징, 커밋, amend, rebase, reset, push처럼 Git 상태나 기록을 바꾸는 작업을 하지 않습니다.

커밋 메시지 추천은 Git 실행 권한이 아니라 최종 보고의 일부입니다. 파일이 바뀌었고 설정에서 커밋 메시지 추천이 켜져 있으면 에이전트는 권장 메시지를 제안할 수 있습니다. 단, 실제 커밋을 만들었다고 표현하면 안 됩니다.

## 버전 영향 정책

`.mustflow/config/preferences.toml`의 `[release.versioning]`은 버전 영향 선호값을 정합니다. 에이전트가 저장소의 실제 버전 기준 원본을 찾은 뒤 버전 파일을 직접 수정할 수 있는지도 여기서 정합니다.

코드, 템플릿, 스키마, CLI 동작, 패키지 메타데이터, 사용자에게 보이는 문서, 설치 출력, 테스트가 바뀌면 에이전트는 패키지나 템플릿 버전 변경이 필요한지 확인해야 합니다.

기본값에서는 근거가 분명하고 버전 기준 원본을 찾았으며 `auto_bump = true`, `require_user_confirmation = false`이면 패치, 마이너, 메이저 판올림을 바로 적용할 수 있습니다. 사용자 직접 지시, 호스트 안전 규칙, 승인 정책이 있으면 자동 버전 수정은 막힙니다.

버전 변경을 제안하거나 적용하기 전에는 `package.json`이라고 가정하지 말고 저장소의 실제 버전 기준 원본을 찾아야 합니다. 자주 확인하는 후보에는 `package.json`, `pyproject.toml`, `Cargo.toml`, 릴리스 태그와 함께 보는 `go.mod`, `pom.xml`, `build.gradle`, `*.csproj`, `*.gemspec`, `composer.json`, `pubspec.yaml`, `Package.swift`, `Chart.yaml`, 앱 매니페스트, 릴리스 노트, mustflow 템플릿 매니페스트가 있습니다.

버전을 바꿀 때는 `sync_*` 선호값에 따라 패키지 메타데이터, 템플릿 매니페스트 버전, 문서 예시, 테스트를 함께 맞춥니다.
