---
title: 지침 새로고침
description: mustflow가 프로젝트 파일의 세션 카운터 대신 새로고침 지점을 쓰는 이유입니다.
---

긴 에이전트 세션에서는 작업 시작 때 읽은 지침이 흐려질 수 있습니다. 도구 출력, 큰 변경
차이, 맥락 압축, 하위 저장소 이동 때문에 처음 읽은 `AGENTS.md`가 뒤쪽 문맥에 밀릴 수
있습니다.

mustflow는 이 문제를 지침 새로고침 지점으로 다룹니다.

## 해결하는 문제

- 위험한 작업 전 에이전트가 필요한 지침 파일을 다시 읽을 수 있습니다.
- 명령 실행 전 기억에 의존하지 않고 `commands.toml`을 다시 확인할 수 있습니다.
- 작업 루트가 바뀌면 가장 가까운 `AGENTS.md`를 다시 읽게 할 수 있습니다.
- 최종 보고 전 보고 규칙을 다시 확인할 수 있습니다.

## 피하는 방식

mustflow는 대화 횟수, 메시지 수, 세션 활동 상태를 프로젝트 파일에 쓰지 않습니다.

그런 상태 파일은 Git 변경을 계속 만들고, 여러 에이전트가 동시에 작업할 때 충돌하며,
불필요한 활동 메타데이터를 남깁니다. 에이전트 호스트가 세션 길이를 추적해야 한다면
로컬 캐시나 호스트가 관리하는 저장소를 사용해야 합니다.

## 새로고침 수준

- `light`: `AGENTS.md`와 `agent-workflow.md`를 다시 읽습니다.
- `command`: `AGENTS.md`와 `commands.toml`을 다시 읽습니다.
- `skill`: `AGENTS.md`와 `skills/INDEX.md`를 다시 읽습니다.
- `full`: mustflow 필수 읽기 순서 전체를 다시 읽습니다.

기준 원본은 `.mustflow/config/mustflow.toml`의 `[refresh]`입니다.

## CLI 방향

나중에 `mf orient`, `mf refresh` 같은 명령이 이 정책을 기계가 읽을 수 있는 계획으로
보여줄 수 있습니다. 현재 템플릿은 모든 도구에 같은 생명주기 훅이 있다고 가정하지 않고,
먼저 정책과 문서 기준을 둡니다.
