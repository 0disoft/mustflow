---
title: manifest.lock.toml 구조 결정
description: mustflow가 manifest.lock.toml 해시 필드를 아직 확장하지 않는 이유입니다.
---

현재 mustflow는 `manifest.lock.toml`에 `content_hash` 하나만 둡니다.

이 값은 “현재 해시”가 아니라 “마지막 설치 또는 갱신 당시 파일 내용 해시”입니다. 이름은 단순하지만 의미는 설치 기준선입니다.

## 결정

현재는 `installed_hash`, `template_hash`, `current_hash`로 나누지 않습니다.

대신 다음 규칙을 사용합니다.

- `content_hash`: 잠금 파일에 저장하는 설치 기준선입니다.
- 현재 파일 해시: 명령 실행 시 파일 시스템에서 계산합니다.
- 현재 템플릿 해시: 명령 실행 시 패키지 안의 템플릿에서 계산합니다.

## 이유

잠금 파일은 재현 가능한 설치 상태만 기록해야 합니다.

`current_hash`는 파일이 바뀔 때마다 달라지는 실행 시점 값입니다. 잠금 파일에 넣으면 사용자가 파일을 수정할 때마다 잠금 파일도 같이 갱신해야 해서 기준선 역할이 흐려집니다.

`template_hash`는 현재 설치된 mustflow 패키지의 템플릿에서 계산할 수 있습니다. 패키지를 바꾸면 템플릿 해시도 바뀌므로, 잠금 파일 안에 오래된 템플릿 해시를 함께 저장하면 두 기준이 충돌할 수 있습니다.

## update 판단 방식

`mf update --dry-run`은 다음 비교만으로 충분합니다.

```text
current file hash == lock content_hash
current file hash == bundled template hash
```

- 첫 번째 비교가 거짓이면 로컬 변경입니다.
- 첫 번째 비교가 참이고 두 번째 비교가 거짓이면 템플릿 갱신 후보입니다.
- 둘 다 참이면 변경할 필요가 없습니다.

## 나중에 확장할 수 있는 경우

다음 요구가 생기면 스키마 판 번호를 올리고 필드를 추가할 수 있습니다.

- 여러 템플릿 원본을 동시에 비교해야 하는 경우
- 템플릿 파일별 원본 해시를 오프라인으로 검증해야 하는 경우
- mustflow 패키지 없이도 갱신 계획을 재현해야 하는 경우
- 서명된 템플릿이나 공급망 검증이 필요한 경우

그 전까지는 `content_hash` 하나를 설치 기준선으로 유지하는 편이 단순하고 안전합니다.
