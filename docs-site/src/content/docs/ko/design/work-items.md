---
title: 작업 항목
description: mustflow에서 로컬 작업 항목을 기본 템플릿에 넣지 않는 이유와 선택형 방향입니다.
---

mustflow는 기본 템플릿에 로컬 이슈나 변경 제안 폴더를 만들지 않습니다.

미해결 작업을 파일로 남기는 기능은 유용할 수 있지만, 기본 설치에 포함하면 mustflow가 에이전트 문서 흐름에서 로컬 이슈 추적기로 확장됩니다. 그래서 현재는 `.mustflow/config/mustflow.toml`에 `work_items = "disabled"`와 `handoff.mode = "report_only"`만 선언합니다.

## 기본값

```toml
[capabilities]
work_items = "disabled"

[handoff]
enabled = false
mode = "report_only"
```

이 설정은 에이전트가 해결하지 못한 일을 새 파일로 쌓지 말고 최종 보고에 남기라는 뜻입니다.

## 기본 템플릿에 넣지 않는 이유

- `mf init`의 기본 목적은 LLM 전용 작업 흐름 설치입니다.
- 로컬 이슈 파일은 오래 방치되기 쉽고, 실제 이슈 추적 도구와 중복될 수 있습니다.
- 실패 로그, 내부 경로, 고객명, 비밀정보 조각이 문서에 섞일 위험이 있습니다.
- 에이전트가 임의로 작업을 만들고 닫기 시작하면 사람의 결정 경계가 흐려집니다.

## 선택형 방향

나중에 선택 기능으로 도입한다면 이름은 `.mustflow/pr/`보다 `.mustflow/work-items/`가 낫습니다. 로컬 문서는 실제 풀 리퀘스트가 아니라 작업 후보와 변경 제안이기 때문입니다.

```text
.mustflow/
└─ work-items/
   ├─ README.md
   ├─ issues/
   │  └─ MF-0001.md
   └─ proposals/
      └─ MF-0001-P001.md
```

`issues/`는 나중에 처리할 버그, 작업, 기능 제안입니다. `proposals/`는 특정 이슈를 해결하기 위한 변경 제안입니다. 실제 브랜치, 코드 차이, 리뷰, 병합은 Git과 협업 플랫폼의 역할입니다.

## 에이전트 권한

선택형 작업 항목을 켜더라도 기본 권한은 좁아야 합니다.

- 에이전트는 문제 후보와 변경 제안을 작성할 수 있습니다.
- 에이전트는 사람 승인 없이 이슈를 닫거나 제안을 승인하면 안 됩니다.
- 에이전트는 실제 풀 리퀘스트가 만들어졌다고 표현하면 안 됩니다.
- 에이전트는 비밀정보, 고객정보, 긴 실패 로그를 작업 항목에 남기면 안 됩니다.

## 나중 명령 후보

```sh
mf work list
mf work show MF-0001
mf work propose MF-0001
mf work check
```

이 명령들은 현재 구현 범위가 아닙니다. 먼저 파일 기반 문서 흐름, 명령 계약, 검증 흐름을 안정화한 뒤 선택형 기능으로 검토합니다.
