title: 작업 항목
description: 선택형 로컬 작업 항목이 mustflow를 확장하면서도 제한된 인계 기록을 유지하는 방식입니다.
---

작업 항목은 저장소 안에 미뤄 둔 이슈, 변경 제안, 재시작 지점을 남기는 선택형 mustflow 표면입니다.

기본 템플릿은 프로젝트가 제한된 생명주기를 선택하기 전까지 `work_items = "disabled"`, `handoff.mode = "report_only"`로 이 표면을 비활성 상태로 둡니다.

## 기본값

```toml
[capabilities]
work_items = "disabled"

[handoff]
enabled = false
mode = "report_only"
```

이 설정은 에이전트가 해결하지 못한 일을 새 파일로 쌓지 말고, 최종 보고에 남기라는 뜻입니다.

## 기본값을 비활성으로 두는 이유

- 기본 설치는 프로젝트가 작업 항목 생명주기를 선택하기 전까지 작게 유지되어야 합니다.
- 로컬 이슈 파일은 오래 방치되기 쉽고, 실제 이슈 추적 도구와 중복될 수 있습니다.
- 실패 로그, 내부 경로, 고객명, 비밀정보 조각이 문서에 섞일 위험이 있습니다.
- 에이전트가 임의로 작업을 만들고 닫기 시작하면 사람의 결정 경계가 흐려집니다.

## 방향

작업 항목 쓰기를 켜면 이름은 `.mustflow/pr/`보다 `.mustflow/work-items/`가 더 적합합니다. 로컬 문서는 실제 풀 리퀘스트가 아니라 작업 후보와 변경 제안 문서이기 때문입니다.

```text
.mustflow/
└─ work-items/
   ├─ README.md
   ├─ issues/
   │  └─ MF-0001.md
   └─ proposals/
      └─ MF-0001-P001.md
```

`issues/`는 나중에 처리할 버그, 작업, 기능 제안을 담습니다. `proposals/`는 특정 이슈를 해결하기 위한 변경 제안을 담습니다. 실제 브랜치, 코드 변경, 리뷰, 병합은 Git과 협업 플랫폼의 역할입니다.

## 에이전트 권한

선택형 작업 항목을 켜더라도 기본 권한은 좁게 가져가야 합니다.

- 에이전트는 문제 후보와 변경 제안을 작성할 수 있습니다.
- 에이전트는 사람 승인 없이 이슈를 닫거나 제안을 승인하면 안 됩니다.
- 에이전트는 실제 풀 리퀘스트가 만들어졌다고 표현하면 안 됩니다.
- 에이전트는 비밀정보, 고객정보, 긴 실패 로그를 작업 항목에 남기면 안 됩니다.

## 명령 후보

```sh
mf work list
mf work show MF-0001
mf work propose MF-0001
mf work check
```

작성과 생명주기 명령은 제한된 스키마, 명령 계약, 비밀정보 가림, 사람 승인 규칙을 갖춘 뒤 점진적으로 추가해야 합니다.
