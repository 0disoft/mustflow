# 웹 이미지 자산 최적화

## 목적

프로젝트에 없는 이미지 처리 파이프라인을 임의로 만들지 않으면서, 웹 이미지 자산을 작고 알맞은 형식으로 유지하고 참조 경로까지 정확히 맞춥니다.

## 사용 조건

- 생성된 PNG, JPEG, WebP, AVIF, GIF 같은 래스터 이미지가 웹사이트나 웹 앱에 추가될 때.
- 기존 웹 이미지를 교체, 크기 조정, 압축 또는 형식 변환할 때.
- 페이지, 컴포넌트, 스타일시트, 메타데이터 파일의 이미지 참조가 바뀔 때.
- 사용자가 이미지 용량 축소, WebP 또는 AVIF 변환, 생성 이미지의 배포 준비를 요청할 때.

## 사용하지 않는 경우

- 대상이 SVG, 아이콘 폰트, 디자인 토큰 파일 같은 벡터 원본일 때.
- 원본 바이트 보존이 웹 전송 용량보다 중요한 보관용, 인쇄용, 증거용 자산일 때.
- 저장소에 더 구체적인 미디어, 디자인 시스템, 프레임워크 자산 절차가 이미 있을 때.
- 새 이미지를 생성만 하고 저장소에 설치하거나 참조하지 않는 작업일 때.

## 필요한 입력

- 이미지 파일 경로와 현재 또는 예정된 참조 위치.
- 예상 표시 크기, 투명도 필요 여부, 애니메이션 필요 여부, 품질 요구 사항.
- 기존 프레임워크 이미지 규칙, 정적 자산 폴더, 가져오기 방식.
- `.mustflow/config/commands.toml`의 `asset_optimize` 및 `build` 의도.

## 사전 조건

- 작업이 사용 조건에 맞고 사용하지 않는 경우에는 해당하지 않습니다.
- 필요한 입력을 확보했거나, 빠진 입력을 추측하지 않고 보고할 수 있습니다.
- 현재 범위에 대해 더 높은 우선순위의 지침과 `.mustflow/config/commands.toml`을 확인했습니다.

## 허용 수정 범위

- 이 스킬, 사용자 요청, `.mustflow/skills/INDEX.md`의 맞는 경로가 설명하는 범위 안에서만 수정합니다.
- 명령 권한을 넓히거나, 프로젝트 사실을 지어내거나, 관련 없는 워크플로 파일을 변경하지 않습니다.

## 절차

1. 변경된 이미지 자산과 그 자산을 참조하는 페이지, 컴포넌트, 스타일, 메타데이터, 매니페스트를 모두 확인합니다.
2. 자산을 바꾸기 전에 원본 파일 크기, 이미지 크기, 형식, 투명도, 애니메이션 여부를 확인합니다.
3. 기존 이미지 파이프라인, 프레임워크 이미지 컴포넌트, 설정된 `asset_optimize` 의도가 있으면 그것을 우선 사용합니다.
4. 사용자가 명시적으로 승인했거나 저장소가 이미 쓰고 있는 경우가 아니라면 새 의존성, 온라인 서비스, 임의 변환 도구를 추가하지 않습니다.
5. 실제 필요에 맞춰 출력 형식을 고릅니다. 대부분의 사진형 또는 생성형 비트맵 웹 자산은 WebP 또는 AVIF를 우선 검토하고, 무손실 투명도나 정확한 픽셀이 중요하면 PNG를 유지하며, 실제 벡터 아트워크에만 SVG를 사용합니다.
6. 저장소에 원본과 파생 자산을 분리하는 규칙이 있으면 따릅니다. 그런 규칙이 없다면 이후 편집에 꼭 필요한 경우가 아니면 중복 원본을 추가하지 않습니다.
7. 지역 스타일에 맞춰 가져오기 구문, 정적 경로, 메타데이터 참조, 치수, 캐시에 민감한 파일 이름을 갱신합니다.
8. 변경 전후 파일 크기를 비교하고, 품질, 투명도, 애니메이션, 브라우저 지원, 대체 형식 관련 위험을 기록합니다.

## 사후 조건

- 명확한 근거, 실행한 명령 의도, 건너뛴 확인, 남은 위험을 포함해 예상 출력을 작성할 수 있습니다.
- 빠진 명령 의도, 알 수 없는 입력, 권한 충돌은 숨기지 않고 보고합니다.

## 검증

가능한 경우 설정된 일회성 명령 의도를 사용합니다.

- `asset_optimize`
- `build`

이미지 최적화 명령을 추측하지 마십시오. `asset_optimize`가 unknown, missing, manual-only 상태이거나 에이전트 실행 가능 상태가 아니라면, 그 상태를 보고하고 설정되지 않은 도구를 실행하지 않은 채 의도한 최적화 내용을 설명합니다.

## 실패 대응

- 최적화 결과가 눈에 띄게 나빠지거나 투명도가 깨지면 더 나은 자산을 복구하거나 유지하고 그 절충점을 보고합니다.
- 이미지 참조가 깨지면 해당 참조를 소유한 가장 가까운 가져오기 구문, 경로, 메타데이터, 빌드 설정을 수정합니다.
- 저장소에 최적화 명령이 없으면 외부 웹사이트나 추측한 패키지 명령을 실행하지 말고, 가장 작은 프로젝트별 명령 의도 추가안을 제안합니다.
- 자산 변경 뒤 빌드가 실패하면 관련 없는 코드를 만지기 전에 이미지 자산과 관련된 첫 실패를 조사합니다.

## 출력 형식

- 검토하거나 변경한 이미지
- 형식 및 용량 변경
- 갱신한 참조
- 실행한 명령 의도
- 건너뛴 명령 의도와 이유
- 남은 이미지 품질 또는 브라우저 지원 위험
