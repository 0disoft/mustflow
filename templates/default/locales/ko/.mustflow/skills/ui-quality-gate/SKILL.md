# UI 품질 게이트

<!-- mustflow-section: purpose -->
## 목적

사용자에게 보이는 인터페이스가 장식용 데모나 깨지기 쉬운 스크린샷처럼 흐르지 않고, 실제로 쓸 수 있고, 간결하고, 반응형이며, 검증 가능하도록 합니다.

<!-- mustflow-section: use-when -->
## 사용할 때

- 변경이 대시보드, 설정, 내비게이션, 폼, 표, 목록, 탭, 버튼, 빈 상태, 툴팁, 상태 메시지, 사용자에게 보이는 문구를 건드립니다.
- 작업이 UI 다듬기, 레이아웃, 반응형 동작, 접근성, 시각 상태, 언어 전환, 라벨, 상호작용 피드백을 요청합니다.
- 보고가 UI 문구가 맞게 들어감, 컨트롤을 이해할 수 있음, 언어 변경이 적용됨, 페이지가 올바르게 렌더링됨을 주장합니다.
- 변경이 설명용, 마케팅식, 장식용, 중복, 지어낸, 행동으로 이어지지 않는 UI 내용을 추가할 수 있습니다.

<!-- mustflow-section: do-not-use-when -->
## 사용하지 않을 때

- 작업이 사용자에게 보이는 UI 표면 없이 백엔드 로직, CLI 출력, 메타데이터, 문서만 바꿉니다.
- 작업이 이미지 자산 변환만 다룹니다. 그 부분에는 `web-asset-optimization`을 사용합니다.
- 현재 환경에서 UI를 렌더링하거나 검사할 수 없습니다. 시각 검증을 했다고 주장하지 말고 검사 공백을 보고합니다.

<!-- mustflow-section: required-inputs -->
## 필요한 입력

- 변경된 UI 표면, 사용자 작업, 기대 상호작용 경로.
- 같은 영역의 기존 디자인 패턴, 라벨, 상태, 접근성 관례, 다국어 규칙.
- 검사해야 하는 화면 크기, 테마, 언어, 상태 조합.
- 상태, diff, 문서, 빌드, 릴리스, mustflow 검증에 관한 명령 의도 계약 항목.

<!-- mustflow-section: preconditions -->
## 전제 조건

- 작업이 사용할 때 조건에 맞고 사용하지 않을 때 조건에는 해당하지 않습니다.
- 필요한 입력이 있거나, 부족한 입력을 추측하지 않고 보고할 수 있습니다.
- 현재 범위의 상위 지시와 `.mustflow/config/commands.toml`을 확인했습니다.

<!-- mustflow-section: allowed-edits -->
## 허용되는 수정

- 사용자의 실제 작업을 돕는 경우 UI 컨트롤, 라벨, 상태, 레이아웃 제약, 다국어 연결, 접근성 속성을 추가, 제거, 다듬습니다.
- 사용자가 실제 데이터에 행동하는 데 도움이 되지 않는 장식용, 설명용, 지어낸, 마케팅식 UI 내용을 제거합니다.
- 새 시각 체계보다 기존 컴포넌트 패턴과 안정적인 크기 제약을 우선합니다.
- 실제 렌더링, 스크린샷, DOM 검사, 또는 시각 검증을 하지 못한 명확한 이유 없이 UI를 시각적으로 검증했다고 주장하지 않습니다.

<!-- mustflow-section: procedure -->
## 절차

1. 실제 사용자 작업과 그 작업을 지원하는 UI 표면을 식별합니다.
2. 새 레이아웃, 컴포넌트, 색, 문구, 상태 관례를 추가하기 전에 주변 UI 패턴을 확인합니다.
3. 필수가 아닌 환영 문구, 기능 요약, 장식 카드, 가짜 지표, 마케팅 문구, 지어낸 컨트롤을 제거하거나 피합니다.
4. 컨트롤을 이해할 수 있는지 확인합니다. 아이콘 버튼에는 접근 가능한 이름이나 툴팁이 필요하고, 파괴적이거나 상태를 바꾸는 동작에는 명확한 라벨이 필요하며, 비활성 상태는 유용할 때 이유가 보여야 합니다.
5. 레이아웃 제약을 확인합니다. 텍스트가 넘치거나 겹치거나, 고정 형식 컨트롤을 예상 밖으로 키우거나, 화면 너비에 따른 글자 크기 조절에 의존하지 않아야 합니다.
6. 상태 범위를 확인합니다. 로딩, 빈 상태, 오류, 저장됨, 변경됨, 비활성, 선택됨, 포커스, 언어 전환 상태가 해당되는 곳에서 일관되게 갱신되어야 합니다.
7. 변경이 레이아웃이나 번역 문구에 영향을 준다면 반응형과 다국어에 민감한 표면을 검사합니다.
8. 변경된 UI, 문서, 패키지, mustflow 계약을 검증하는 가장 좁은 구성된 검증을 실행합니다.

<!-- mustflow-section: postconditions -->
## 완료 조건

- UI가 불필요한 설명용 또는 장식용 표면 없이 사용자의 작업을 지원합니다.
- 중요한 컨트롤, 라벨, 상태, 레이아웃 제약, 다국어 갱신을 확인했거나 미검증으로 보고했습니다.
- 최종 보고는 코드 수준 검증과 시각 또는 상호작용 검증을 구분합니다.

<!-- mustflow-section: verification -->
## 검증

사용 가능한 구성된 일회성 명령 의도를 사용합니다.

- `changes_status`
- `changes_diff_summary`
- `docs_validate_fast`
- `test_release`
- `mustflow_check`

변경한 UI 표면을 더 잘 증명하는 좁은 테스트, 빌드, 브라우저, 스크린샷, 접근성 의도가 있으면 그 의도를 사용합니다.

<!-- mustflow-section: failure-handling -->
## 실패 처리

- 시각 검사를 할 수 없다면 동작한다고 가정하지 말고 검증하지 못한 화면 크기, 상태, 상호작용을 보고합니다.
- UI 텍스트가 겹치거나 잘리거나 상태 또는 언어 변경 뒤 갱신되지 않으면 더 넓은 레이아웃 변경 전에 가장 작은 소유 컴포넌트를 고칩니다.
- 요청된 UI 요소가 저장소의 UI 미니멀리즘 규칙과 충돌하면 작업 중심의 가장 작은 컨트롤을 구현하고 생략한 장식 요소를 보고합니다.
- 검증에 명령 계약에 선언되지 않은 대화형 서버나 브라우저 명령이 필요하면 그 경계에서 멈추고 건너뛴 검사를 보고합니다.

<!-- mustflow-section: output-format -->
## 출력 형식

- 검토한 UI 표면
- 확인한 사용자 작업과 상태
- 레이아웃, 접근성, 다국어 확인
- 피하거나 제거한 장식용 또는 불필요한 UI
- 실행한 명령 의도
- 건너뛴 시각 검사와 이유
- 남은 UI 위험
