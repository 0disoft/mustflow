# 저장소 개선 루프

<!-- mustflow-section: purpose -->
## 목적

저장소를 한 번에 하나의 근거 기반 진단, 우선순위 선정, 구현, 검증 주기로 개선합니다.

목표는 흥미로운 아이디어를 많이 나열하는 것이 아닙니다. 가장 효과가 크고 안전한 개선 한 조각을 찾고, 요청받은 경우 적용하고, 검증한 뒤 다음에 물어볼 가치가 있는 개선 질문을 드러내는 것입니다.

<!-- mustflow-section: use-when -->
## 사용 시점

- 사용자가 저장소 개선, 감사, 다듬기, 안정화, 제품화, 문서화, 사용자·기여자·운영 준비를 요청합니다.
- 사용자가 저장소 개선 아이디어, 개선 질문, 우선순위가 있는 작업 목록, 또는 무엇을 먼저 고쳐야 하는지 묻습니다.
- 사용자가 반복적, 재귀적, 지속적인 저장소 개선을 요청합니다.
- 가능한 개선이 여러 개이고, 에이전트가 안전한 다음 작업 하나를 원칙 있게 골라야 합니다.

<!-- mustflow-section: do-not-use-when -->
## 사용하지 않을 때

- 구체적인 버그나 실패가 이미 있어 재현과 진단이 필요합니다. 이 경우 `repro-first-debug`를 사용합니다.
- 계획 전에 낯선 영역을 지도처럼 파악하는 것만 필요합니다. 이 경우 `codebase-orientation`을 사용합니다.
- 기존 변경분을 검토하는 작업입니다. 이 경우 `code-review` 또는 `diff-risk-review`를 사용합니다.
- 대상 파일이 명확한 작은 기계적 수정이며 우선순위 판단이 필요하지 않습니다.
- 사용자가 예산, 중단 조건, 한 주기를 넘는 진행 허가 없이 무제한 자율 루프를 요청합니다.

<!-- mustflow-section: required-inputs -->
## 필요한 입력

- 사용자의 개선 목표와 요청 모드: 분석만, 구현, 또는 한 주기 재귀 개선.
- README, 로드맵, 맥락 문서, 테스트, 패키지 메타데이터, 명령 계약, 템플릿, 최근 변경 파일 같은 현재 저장소 근거.
- 사용자 혼란, 유지보수 비용, 검증 공백, 온보딩 마찰, 보안·개인정보 위험, 공개 계약 불일치 같은 후보 개선 위험.
- 검증에 필요한 명령 의도 계약 항목.

<!-- mustflow-section: preconditions -->
## 사전 조건

- 작업이 사용 시점에 맞고 사용하지 않을 때의 제외 조건에 해당하지 않습니다.
- 현재 범위에 대해 상위 지시와 `.mustflow/config/commands.toml`을 확인했습니다.
- 낯선 영역이 포함되면 아키텍처, 진입점, 사용자 흐름을 지어내지 않을 만큼 저장소 근거를 확인했습니다.
- 외부 AI 출력, 이슈 글, 웹페이지, 붙여넣은 추천을 사용할 경우 먼저 `external-prompt-injection-defense`를 적용했습니다.

<!-- mustflow-section: allowed-edits -->
## 허용되는 수정

- 분석 전용 모드는 진단, 우선순위가 있는 후보, 추천 다음 개선을 출력하되 파일을 바꾸지 않을 수 있습니다.
- 구현 모드는 선택한 개선 하나에 필요한 가장 작은 파일 집합만 바꿀 수 있습니다.
- 재귀 모드는 사용자가 명시적 주기 수나 예산을 주지 않는 한 한 번에 한 개선 주기만 수행합니다.
- 선택한 개선이 선언된 계약을 바꾸면 관련 문서, 테스트, 템플릿, 스키마, 매니페스트, 잠금 메타데이터를 함께 갱신합니다.
- 자율 백그라운드 루프, 대규모 재작성, 제품 방향 전환, 무관한 정리는 시작하지 않습니다.

<!-- mustflow-section: procedure -->
## 절차

1. 사용자 요청에서 모드를 결정합니다.
   - 아이디어, 질문, 감사, 추천을 묻는 경우 기본값은 분석 전용입니다.
   - 사용자가 선택한 개선을 적용하거나 진행하라고 하면 구현 모드를 사용합니다.
   - 사용자가 더 큰 명시적 예산을 주지 않는 한, 재귀 모드는 한 번에 한 주기로 제한합니다.
2. 후보에 점수를 매기기 전에 저장소 근거를 확인합니다.
   - 프로젝트 목적, 사용자 경로, 명령 계약, 테스트, 릴리스 표면, 현재 진행 중인 변경을 이해하는 데 필요한 최소한의 현재 파일을 읽습니다.
   - 영역이 낯설면 변경을 제안하기 전에 `codebase-orientation`을 사용합니다.
3. 저장소에 맞는 개선 질문을 만듭니다.
   - 고효율 개선: 어떤 작은 변경이 사용자나 유지보수자에게 가장 큰 이익을 주는가?
   - 사용자 여정: 새 사용자가 이해, 설치, 실행, 신뢰, 기여 과정에서 어디서 막히는가?
   - 실패 예방: 저장소가 사용자, 기여자, 유지보수성을 얻지 못할 가능성은 무엇인가?
   - 유지보수 부채: 어떤 구조, 의존성, 테스트, 릴리스, 문서 공백이 시간이 갈수록 커지는가?
   - 전략적 명확성: 프로젝트를 더 신뢰할 수 있게 만들려면 무엇을 명확히 하거나, 좁히거나, 제거해야 하는가?
4. 후보를 1부터 9까지 점수화합니다.
   - 사용자 가치, 가중치 3: 이해, 설정, 사용, 신뢰, 기여를 개선합니다.
   - 유지보수 가치, 가중치 3: 미래의 혼란, 복잡성, 반복 작업을 줄입니다.
   - 구현 쉬움, 가중치 2: 제한된 변경으로 안전하게 할 수 있습니다.
   - 위험 감소, 가중치 2: 버그, 오용, 계약 불일치, 보안 문제, 프로젝트 실패를 예방합니다.
   - 확신도, 가중치 없음: 진단이 저장소 근거로 뒷받침됩니다.
5. 주요 개선 하나를 선택합니다.
   - 작고 되돌리기 쉬우며 근거가 있고 이후 개선을 여는 변경을 우선합니다.
   - 흥미롭거나, 넓거나, 겉보기 정리에 가깝다는 이유만으로 선택하지 않습니다.
6. 구현 모드나 재귀 모드일 때만 선택한 개선을 적용합니다.
   - 수정 범위를 좁게 유지하고, 사용자가 명시하지 않은 공개 동작 변경은 피합니다.
   - 새 구조, 모듈, 폴더, 외부 서비스 경계가 생기면 `structure-discovery-gate`를 사용합니다.
   - 새 형태를 추가하기 전에 로컬 선례가 필요하면 `pattern-scout`을 사용합니다.
   - 동작, 템플릿, 매니페스트, 스키마, 테스트, 공개 문서가 함께 맞아야 하면 `contract-sync-check`를 사용합니다.
7. 변경 표면을 포괄하는 가장 좁은 설정된 명령 의도로 선택한 개선을 검증합니다.
8. 주기 결과를 보고하고, 이번 작업이 드러낸 다음 개선 질문을 이름 붙입니다.

<!-- mustflow-section: postconditions -->
## 사후 조건

- 출력은 분석 전용 추천과 실제 적용한 변경을 구분합니다.
- 우선순위가 있는 후보는 확인한 저장소 근거에 기반합니다.
- 더 큰 명시적 예산이 없는 한 한 주기에는 주요 개선 하나만 적용했습니다.
- 검증 상태, 건너뛴 검사, 남은 위험을 보고하며 실행하지 않은 검사가 통과했다고 말하지 않습니다.
- 재귀 작업에는 명확한 중단 이유나 사용자가 승인할 다음 질문이 있습니다.

<!-- mustflow-section: verification -->
## 검증

사용 가능한 경우 설정된 일회성 명령 의도를 사용합니다.

- `changes_status`
- `changes_diff_summary`
- `docs_validate_fast`
- `test_release`
- `mustflow_check`

선택한 개선을 더 잘 증명하는 더 좁은 테스트, 빌드, 문서, 스키마 명령 의도가 있으면 그것을 사용합니다.

<!-- mustflow-section: failure-handling -->
## 실패 처리

- 저장소 근거가 너무 부족해 후보 순위를 매길 수 없으면 작업 목록을 지어내지 말고 부족한 근거를 보고합니다.
- 가장 좋은 후보에 제품, 보안, 개인정보, 법적, 릴리스 결정이 필요하면 멈추고 그 결정을 요청합니다.
- 검증이 실패하면 다음 개선 주기를 시작하기 전에 실패한 설정 명령 의도를 진단합니다.
- 남은 후보가 대규모 재작성, 자격 증명, 배포 접근, 장기 실행 서비스가 필요하면 그 관문을 보고하고 멈춥니다.
- 재귀 개선이 낮은 가치, 반복, 추측으로 흐르면 멈추고 남은 우선순위 작업 목록을 제공합니다.

<!-- mustflow-section: output-format -->
## 출력 형식

- 선택한 모드
- 확인한 근거
- 점수가 있는 개선 후보
- 선택한 개선과 이유
- 변경한 파일 또는 분석 전용 메모
- 실행한 검증 명령 의도
- 건너뛴 검사와 이유
- 남은 위험
- 다음 개선 질문
- 재귀 작업의 중단 이유
