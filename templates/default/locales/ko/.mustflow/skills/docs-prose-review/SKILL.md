# 문서 문장 검수

<!-- mustflow-section: purpose -->
## 목적

검수 대기열의 문서 하나를 기술적 계약은 유지한 채 자연스러운 문장으로 다듬습니다.

<!-- mustflow-section: use-when -->
## 사용 조건

- LLM이 만들었거나 수정한 문서를 검수하라는 요청을 받았습니다.
- 문서가 mustflow 문서 검수 대기열에 올라와 있습니다.
- 대기열 항목에 문서를 어떻게 확인하거나 수정하면 좋을지 설명하는 검수 코멘트가 붙어 있습니다.
- 문장이 LLM이 쓴 듯하거나, 직역투이거나, 군더더기가 많거나, 설명이 중복되거나, 한국어/영어/현지화 문장이 부자연스럽습니다.
- 검수자가 사람, LLM, 도구, 외부 프로세스 중 하나이며 검수 결과를 기록해야 합니다.

<!-- mustflow-section: do-not-use-when -->
## 사용하지 않는 경우

- 문장 검수가 아니라 새 제품 문서, API 문서, 워크플로 정책 변경을 요청받았습니다.
- 문서가 검수 대기열에 없고, 사용자가 그 문서를 대기열에 추가하라고 요청하지 않았습니다.
- 요청된 변경이 명령어, 경로, 코드 블록, 스키마, 필드명, 공개 계약, 기술적 의미를 바꿉니다.
- 검수자가 대상 언어를 충분히 이해하지 못해 안전하게 문장을 다듬을 수 없습니다.

<!-- mustflow-section: required-inputs -->
## 필요한 입력

- 문서 검수 대기열 항목 또는 사용자가 고른 문서 경로.
- 대기열 항목에 붙은 검수 코멘트.
- 현재 파일 내용.
- 문서의 대상 언어와 기존 문서 구조.
- 검수 기록에 남길 검수자 종류와 자유 형식 검수자 식별자.
- 검증 의도를 확인하기 위한 `.mustflow/config/commands.toml`.

<!-- mustflow-section: preconditions -->
## 사전 조건

- 작업이 사용 조건에 맞고 사용하지 않는 경우에 해당하지 않습니다.
- 대기열 항목 또는 선택한 경로가 현재 mustflow 루트에 존재합니다.
- 현재 범위에 대해 더 높은 우선순위의 지침, 저장소 스타일, 명령 정책을 확인했습니다.
- 검수자가 문장을 다듬으면서 기술적 의미를 유지할 수 있습니다.

<!-- mustflow-section: allowed-edits -->
## 허용 수정 범위

- 사용자가 명시적으로 범위를 넓히지 않는 한 선택한 문서 파일과 검수 기록 항목만 수정합니다.
- 문장 문제가 직접 거기서 생긴 경우가 아니라면 제목, 앞부분 식별 정보, 표, 명령 예시, 코드 블록, 경로, 필드명, 스키마 이름을 유지합니다.
- 말투만 바꾸려고 문서 전체를 다시 쓰지 않습니다.
- 문서를 개선하거나, 아직 검수가 필요하다고 표시하거나, 무시해야 하는 이유를 명확히 기록하지 않은 채 대기열에서 제거하지 않습니다.

<!-- mustflow-section: procedure -->
## 절차

1. 사용자가 특정 경로를 고르지 않았다면 문서 검수 대기열을 확인하고 대기 중인 파일 하나를 선택합니다.
2. 항목에 검수 코멘트가 있으면 그 내용을 우선 검수 지침으로 삼습니다. 이 스킬의 기술적 안전 경계는 그대로 유지합니다.
3. 항목에 검수 코멘트가 없으면 문서에서 어색하거나, LLM이 쓴 듯하거나, 과하게 설명했거나, 중복됐거나, 직역투이거나, 부자연스러운 문장을 직접 확인합니다.
4. 용어, 제목 구조, 예시, 참조가 일관되도록 수정 전에 선택한 파일 전체를 읽습니다.
5. 검수 코멘트 또는 문장 문제에 따라 최소한으로 수정하고 의미를 유지합니다.
6. 실행 가능한 예시, 경로, 필드명, 옵션명, 식별자, 앞부분 식별 정보, 표는 보존합니다.
7. 코멘트가 모호하거나 의미가 불명확하면 추측하지 않습니다. 해당 항목을 아직 사람 검수가 필요한 상태로 표시하고 사람이 판단해야 할 내용을 요약합니다.
8. 문장 수정이 필요 없는 파일이라면 간결한 이유를 남기고 승인 또는 무시 처리합니다.
9. 문장 검수가 성공적으로 끝나면 검수자 메타데이터와 짧은 요약을 남기고 대기열 항목을 승인 처리합니다.
10. 공개 문서나 설치형 워크플로 문서를 바꿨다면 관련된 구성된 검증 의도를 실행합니다.

<!-- mustflow-section: postconditions -->
## 사후 조건

- 선택한 문서가 기술적 의미를 바꾸지 않은 채 더 자연스럽게 읽힙니다.
- 검수 대기열 항목이 검수자 메타데이터와 함께 승인, 사람 검수 필요, 무시 중 하나로 기록됩니다.
- 건너뛴 수정, 해결되지 않은 의미 질문, 누락된 명령 의도가 보고됩니다.

<!-- mustflow-section: verification -->
## 검증

사용 가능하고 관련이 있을 때 구성된 일회성 명령 의도를 사용합니다.

- `docs_validate`
- `mustflow_check`

누락된 검증 명령은 추측하지 않습니다.

<!-- mustflow-section: failure-handling -->
## 실패 대응

- 대기열을 확인할 수 없으면 막힌 대기열 단계를 보고하고 추측으로 수정하지 않습니다.
- 선택한 파일이 없으면 대체 문서를 만들지 말고 오래된 대기열 항목을 표시하거나 보고합니다.
- 언어나 기술적 의미가 불확실하면 해당 항목을 아직 사람 검수가 필요한 상태로 표시합니다.
- 문장 수정 후 검증이 실패하면 검수 완료로 표시하기 전에 관련 문서 또는 워크플로 문제를 먼저 고칩니다.

<!-- mustflow-section: output-format -->
## 출력 형식

- 선택한 대기열 항목
- 따른 검수 코멘트 또는 코멘트가 없었다는 이유
- 수정한 문장 문제
- 기록한 검수 상태
- 사용한 검수자 종류와 검수자 식별자
- 실행한 명령 의도
- 건너뛴 명령 의도와 이유
- 남은 언어, 의미, 검증 위험
