# 의존성 실재 확인

<!-- mustflow-section: purpose -->
## 목적

코드, 문서, 테스트, 최종 보고가 사용할 수 없는 패키지, 도구, 명령, 런타임, 서비스, 플랫폼 기능을 있다고 가정하지 않게 합니다.

<!-- mustflow-section: use-when -->
## 사용할 때

- 의존성, 도구, 런타임, 명령, 플러그인, 서비스, 플랫폼 기능을 추가, 제거, 이름 변경, 가져오기, 호출, 문서화합니다.
- 해법이 패키지 관리자, 실행 파일, 환경 변수, 브라우저 API, 운영체제 명령, 호스팅 서비스, 선택 통합에 의존합니다.
- 생성한 안내문이 다른 에이전트나 사용자에게 저장소에 선언되지 않은 도구를 실행하라고 말합니다.
- 실패 원인이 누락된 설치, 맞지 않는 버전, 지원되지 않는 런타임, 사용할 수 없는 명령일 수 있습니다.

<!-- mustflow-section: do-not-use-when -->
## 사용하지 않을 때

- 작업이 저장소 내부 문구만 바꾸며 도구, 런타임 동작, 패키지 메타데이터, 명령을 언급하지 않습니다.
- 현재 작업 맥락에서 의존성이 이미 증명되었고 의존성 관련 표면이 바뀌지 않습니다.
- 사용자가 아직 구현하거나 검증하지 않을 추측성 설계만 요청했습니다.

<!-- mustflow-section: required-inputs -->
## 필요한 입력

- 가정하는 의존성, 도구, 명령, 런타임, 서비스, 플랫폼 기능.
- 그것을 선언하거나 참조하는 패키지, 잠금, 설정, 가져오기, 스크립트, 명령 의도, 문서 파일.
- 필요한 경우 최소 버전, 기능, 사용 가능성 주장.
- 빌드, 패키지, 테스트, 문서 검증에 관한 명령 의도 계약 항목.

<!-- mustflow-section: preconditions -->
## 전제 조건

- 작업이 사용할 때 조건에 맞고 사용하지 않을 때 조건에는 해당하지 않습니다.
- 필요한 입력이 있거나, 부족한 입력을 추측하지 않고 보고할 수 있습니다.
- 현재 범위의 상위 지시와 `.mustflow/config/commands.toml`을 확인했습니다.

<!-- mustflow-section: allowed-edits -->
## 허용되는 수정

- 같은 기능을 설명하는 의존성 선언, 가져오기, 스크립트, 명령 메타데이터, 테스트, 문서를 맞춥니다.
- 새 패키지나 도구를 추가하기 전에 저장소의 기존 의존성과 선언된 명령 의도를 우선합니다.
- 사용자 요청과 저장소 계약이 뒷받침하지 않으면 패키지를 설치하거나 런타임 요구사항을 넓히거나 새 외부 서비스를 도입하지 않습니다.
- 인터넷이나 다른 프로젝트에 있다는 이유만으로 의존성이 사용 가능하다고 주장하지 않습니다.

<!-- mustflow-section: procedure -->
## 절차

1. 가정하는 의존성이나 기능과 작업이 그것에 의존하는 지점을 이름 붙입니다.
2. 패키지 메타데이터, 잠금 파일, 설정 파일, 가져오기, 명령 의도, 문서, 템플릿 같은 저장소 선언을 먼저 확인합니다.
3. 의존성이 존재하는지, 없는지, 선택 사항인지, 전이 의존성인지, 호스트 제공인지, 외부 시스템인지 판단합니다.
4. 존재한다면 요청한 기능과 버전 기대값이 선언된 의존성과 맞는지 확인합니다.
5. 없다면 기존 로컬 대안을 우선합니다. 새 의존성은 필요하고 작업 범위 안에 있을 때만 추가합니다.
6. 패키지 메타데이터, 의도적으로 갱신한 잠금 파일, 명령 계약, 문서, 테스트, 설치 안내 같은 의존성 관련 표면을 모두 맞춥니다.
7. 변경이 사용하는 의존성 경로를 증명하는 가장 좁은 구성된 검증을 실행합니다.

<!-- mustflow-section: postconditions -->
## 완료 조건

- 모든 의존성이나 도구 주장은 저장소 선언, 구성된 명령, 호스트 경계, 또는 명시한 미검증 위험으로 뒷받침됩니다.
- 새 의존성 요구사항은 알맞은 메타데이터와 공개 문서에 반영되어 있습니다.
- 최종 보고는 의존성이 기존 항목인지, 추가되었는지, 선택 사항인지, 사용할 수 없는지, 의도적으로 검증하지 않았는지 밝힙니다.

<!-- mustflow-section: verification -->
## 검증

사용 가능한 구성된 일회성 명령 의도를 사용합니다.

- `changes_status`
- `changes_diff_summary`
- `build`
- `test_release`
- `mustflow_check`

의존성 경로를 더 잘 증명하는 좁은 테스트, 패키지, 문서 의도가 있으면 그 의도를 사용합니다.

<!-- mustflow-section: failure-handling -->
## 실패 처리

- 의존성이 없다면 조용히 우회하지 말고 빠진 선언이나 명령을 보고합니다.
- 선언된 버전이 필요한 기능을 제공하지 않는다면 불일치를 보고하고 지원한다고 말하지 않습니다.
- 의존성에 네트워크, 인증 정보, 운영체제 설정, 서비스 접근이 필요하면 그 경계에서 멈추고 확인하지 못한 요구사항을 밝힙니다.
- 생성 문서가 선언되지 않은 도구를 실행하라고 안내한다면 선언된 명령을 쓰도록 고치거나 수동 사전 조건으로 표시합니다.

<!-- mustflow-section: output-format -->
## 출력 형식

- 확인한 의존성 또는 기능
- 저장소 선언 또는 부재
- 맞춘 표면
- 실행한 명령 의도
- 건너뛴 의존성 확인과 이유
- 남은 의존성 위험
